# Java 后端 → Agent 应用研发：24 周学习路线

> 建议投入：每周 12–15 小时。  
> 主项目：**Project Diagnostics Agent（研发排障智能体）**。整个路线只维护一个不断演进的项目，避免做一堆无法证明深度的 Demo。

## 最终交付标准

24 周后，项目应能接收一个“服务启动失败 / 接口异常 / 日志报错”任务，检索项目文档和历史案例，调用只读诊断工具，给出带引用的判断与修复建议；高风险操作必须人工审批。系统包含 Python Agent 服务、Java 业务网关、状态存储、评测集、Tracing、Docker 部署和完整架构说明。

## 前 8 周的具体执行

### 第 1–2 周：Python 工程底座

- 建立 `src/tests` 项目结构，使用类型标注、Pydantic、pytest 和环境变量管理。
- 实现 FastAPI 异步接口；统一错误码、超时、重试和结构化日志。
- 输出一篇“Java 并发/异常模型与 Python asyncio 的差异”学习笔记。

### 第 3–4 周：LLM 与结构化输出

- 封装模型客户端，不把供应商 SDK 散落在业务代码中。
- 实现 JSON Schema/Pydantic 结构化输出、上下文长度控制和 Token 统计。
- 为正常、超时、格式漂移、内容为空和限流五类情况写测试。

### 第 5–6 周：无框架 Agent Loop

- 手写 `observe → plan → act → observe` 循环，再加入最大步数和终止条件。
- 工具先做只读：读取项目文件、查看 Git 状态、检查端口；参数必须经过 Schema 校验。
- 保存每一步轨迹，建立 20 条任务集，统计工具选择正确率和任务完成率。

### 第 7–8 周：RAG 第一版

- 接入项目 README、错误码文档、部署手册和故障案例。
- 比较向量检索与关键词检索，加入重排和带出处回答。
- 建立 50 条问答/故障查询集；记录 Recall@k、答案正确率、P95 延迟和平均 Token。

## 学习纪律

1. 每个新概念必须落到主项目；没有代码、测试或指标的不算完成。
2. 先实现无框架版本，再引入 LangGraph；框架名称不等于能力。
3. 每两周保留一次可运行版本，并记录准确率、延迟、Token 和失败类型。
4. 训练/微调、GPU 推理、大数据、知识图谱和多模态只在目标岗位明确要求时插入路线。
5. 24 周内至少 60% 时间写代码和测试，20% 做评测与复盘，20% 阅读资料。

## 现在开始的第一周

- 建一个 Python 3.12 + FastAPI + Pydantic + pytest 的最小仓库。
- 完成 `/health`、`/diagnose` 两个接口和 10 个测试。

- 本周结束时只回答三个问题：接口能否运行、异常能否被测试、下一步能否安全接入模型 API。
