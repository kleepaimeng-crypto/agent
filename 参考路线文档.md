## 1. 整体方法：把学习设计成闭环

目前的计划已经抓住了两个关键输入：

1. **技术资料**决定“Agent 到底是什么、应该怎么实现”。
2. **企业岗位**决定“哪些能力值得优先投入”。

但如果只有资料和岗位，容易演变成“收藏了一堆内容，却不知道下一步学什么”。因此还需要补上两层：

3. **项目证据层**：学完后用什么作品证明自己会了。
4. **评估反馈层**：如何判断自己真的掌握、哪里需要补、路线是否应该调整。

完整的学习系统应该形成如下循环：

```text
岗位需求
   ↓
能力矩阵
   ↓
学习路线
   ↓
资料与实验
   ↓
项目作品
   ↓
测试与评估
   ↓
重新对照岗位需求
```

这套闭环同时回答五个问题：

- 企业需要什么？
- 自己目前缺什么？
- 应该按什么顺序学习？
- 如何证明已经掌握？
- 如何根据新信息持续调整路线？

## 2. 明确目标岗位方向

“Agent 岗位”实际上混合了几种差异很大的方向。学习前应先明确主线，避免把应用开发、平台工程和算法研究混成一套没有重点的路线。

### 2.1 方向 A：Agent 应用研发 / AI 后端工程师

主要工作包括：

- 调用大模型 API；
- Tool Calling；
- Agent 工作流；
- RAG；
- MCP；
- 数据库与外部系统接入；
- 后端接口开发；
- 部署、日志、监控与评估；
- 把 Agent 做成可以实际使用的产品。

如果已经有 Java、Spring Boot、MySQL、Redis、Docker、Linux、Vue 和 Python 基础，这通常是最适合作为主线的方向。

### 2.2 方向 B：Agent 平台 / Agent Infra 工程师

在应用研发的基础上，更强调：

- 高并发；
- 容器和沙箱；
- Kubernetes；
- 任务调度；
- 分布式系统；
- 长任务恢复；
- 权限管理；
- 可观测性；
- Agent 执行平台。

它适合作为应用研发之后的第二阶段发展方向。

### 2.3 方向 C：Agent 算法工程师

通常要求：

- Transformer 和深度学习基础；
- PyTorch；
- 模型训练、微调；
- 强化学习；
- 推理优化；
- 论文复现；
- Agent 算法研究。

这条路线学习成本更高。如果当前目标是软件开发或大模型应用岗位，不建议一开始就把它作为主线，以免分散工程实践的精力。

### 2.4 初步定位

建议将目标岗位暂定为：

> **主攻 Agent 应用研发 / AI 后端工程师，逐步补充 Agent 平台工程能力。**

该定位不是永久不变的。后续可以根据岗位库中的真实需求、个人兴趣和项目表现定期修正。

## 4. 岗位库的收集与分析

### 4.1 控制第一批样本规模

不要一开始漫无目的地收集几百个岗位。第一版收集 **30～50 个有效岗位** 即可形成初步判断。

建议覆盖：

- Agent 应用研发；
- 大模型应用 / RAG 工程师；
- AI 后端 / Agent 平台工程师；
- Agent 实习生 / 校招岗位。

公司类型尽量多样化：

- 大型互联网公司；
- 中型 AI 公司；
- 普通软件公司的 AI 应用岗位；
- 创业公司；
- 实习、校招和 1～3 年经验岗位。

即使部分岗位暂时要求工作经验，也有参考价值，因为它们可以反映企业真正使用的技术和希望候选人解决的问题。

### 4.2 每个岗位都要结构化保存

不要只保存截图或网页链接。每个岗位至少提取以下字段：

```text
公司：
岗位：
发布日期：
岗位类型：
学历要求：
工作经验：
必须技能：
加分技能：
编程语言：
Agent 技术：
RAG 技术：
后端技术：
数据库：
部署技术：
算法基础：
业务场景：
岗位链接：
个人匹配度：
```

将这些字段统一整理进 `jobs.csv`，才能进行后续的筛选、统计和比较。

### 4.3 不要只统计关键词频率

下面这种统计有参考价值，但不能直接等同于学习优先级：

```text
LangGraph 出现 12 次
MCP 出现 18 次
RAG 出现 25 次
```

原因是不同公司可能使用不同名称描述同一种能力。例如，`LangGraph`、`Agent Workflow` 和“复杂任务编排”背后可能都对应：

```text
有状态工作流
条件分支
任务恢复
人工审批
多步骤执行
```

因此，需要先把岗位关键词归并为能力类别，再统计能力需求。

建议使用以下分类：

```text
编程与计算机基础
LLM 基础
Prompt 与上下文工程
Tool Calling
Agent Loop
RAG 与检索
状态与记忆
工作流编排
MCP 与外部集成
Agent Evaluation
安全与权限
后端与数据库
部署与可观测性
模型训练与算法
```

### 4.4 给岗位要求加权

可以先采用一套简单权重：

```text
目标岗位的必备要求：3 分
目标岗位的加分要求：2 分
相邻岗位的必备要求：1 分
单家公司特有要求：0.5 分
```

再综合计算：

```text
能力优先级 = 岗位出现频率 × 岗位匹配度 × 要求权重
```

这样可以避免因为某个热门框架短期频繁出现，就立刻打乱整条学习路线。

### 4.5 岗位库应产出的结果

岗位库不是资料仓库，而应定期生成：

- 目标岗位画像；
- 技能频率表；
- 同义技能归并表；
- 个人能力差距表；
- 未来两个月学习优先级；
- 需要通过项目证明的能力清单。

## 5. 资料库的建立与使用

### 5.1 避免把资料库变成收藏夹

资料收集最大的问题是：**收集行为很容易代替学习行为。**

找到十门课程、二十个仓库和五十篇文章，不代表已经开始掌握 Agent。资料的价值取决于它是否被转化为理解、实验、测试和项目功能。

每个主题的初始资料建议控制为：

- 一份官方资料；
- 一份容易理解的教程；
- 一个可以运行的项目；
- 必要时再加一篇论文。

以 Tool Calling 为例：

```text
官方文档：理解接口和准确概念
入门教程：帮助快速跑通
示例仓库：阅读工程结构
自己的实验：修改、破坏并观察结果
```

### 5.2 为重要资料建立 Source Card

每份重要资料都建立一张资料卡：

```text
资料名称：
资料类型：官方文档 / 教程 / 仓库 / 论文
来源链接或本地路径：
对应能力：
为什么阅读：
前置知识：
重点章节：
可以跳过的章节：
阅读后的输出：
关联实验：
阅读状态：
可靠性：
最后复核日期：
```

“阅读后的输出”不能只写“读完”，应该写成可以验收的能力，例如：

```text
能解释 Tool Calling 的完整执行过程
能自己定义三个工具
能处理工具参数错误
能记录每次工具调用
能说明 Tool Calling 与普通 API 调用的区别
```

### 5.3 资料优先级

建议遵循：

```text
官方文档
   >
官方示例和维护活跃的开源仓库
   >
成熟团队的技术文章
   >
系统课程
   >
普通博客和短视频
```

对于快速变化的 Agent 框架和 API，还应记录资料的发布日期、适用版本和最后复核日期，避免用过期资料指导当前实现。

### 5.4 资料库应产出的结果

每一批资料都应转化为：

- 资料适用阶段；
- 必读、选读和暂时跳过的内容；
- 推荐阅读顺序；
- 核心概念问题；
- 最小编码实验；
- 尚未回答的问题；
- 对能力矩阵和路线的影响。

## 6. 每个知识点的统一学习方法

以后无论学习 RAG、MCP、LangGraph，还是 Agent Evaluation，都使用相同的五步法。

### 第一步：建立概念问题

不要直接打开教程照着写。先提出问题：

```text
它解决什么问题？
没有它会怎样？
输入和输出是什么？
核心执行流程是什么？
哪些部分由模型决定？
哪些部分由程序决定？
什么时候不应该使用？
常见失败原因是什么？
```

这些问题会帮助你把注意力放在机制上，而不是框架函数名上。

### 第二步：完成最小实现

尽量先不用大型框架，自己实现最小流程。

例如 Agent Loop：

```text
用户输入
→ 调用模型
→ 判断是否调用工具
→ 执行工具
→ 把工具结果交回模型
→ 模型继续判断
→ 输出最终答案
```

如果只会调用一个高级封装函数，却解释不清这个循环，就不能算真正掌握。

### 第三步：使用框架重新实现

最小实现跑通后，再使用框架重写，例如：

- OpenAI Agents SDK；
- LangChain；
- LangGraph。

重点不是重复写同一个 Demo，而是比较：

- 框架替你封装了什么；
- 状态保存在哪里；
- 工具错误如何传播；
- 终止条件如何控制；
- tracing 和评估如何接入；
- 哪些约束仍需要自己负责。

### 第四步：主动制造故障

例如：

- 工具参数错误；
- 工具超时；
- 工具返回空数据；
- 模型重复调用工具；
- RAG 未检索到正确文档；
- Agent 陷入循环；
- 用户拒绝执行危险操作；
- 任务在执行中途被打断。

Agent 学习不能只看成功案例。能够定位和处理失败，才是更有就业价值的能力。

### 第五步：形成可证明的输出

每个知识点至少产出一种可检查的成果：

- 可运行代码；
- 实验报告；
- 架构图；
- 测试数据集；
- 错误案例；
- 技术文章；
- 项目功能。

掌握程度应按以下层级递进：

```text
能解释
→ 能实现
→ 能修改
→ 能调试
→ 能评估
→ 能部署
```

## 7. 完整阶段路线

结合已有后端开发基础，可以设计一条约 **8～10 个月**的主线。时间只是参考，真正的推进依据是每个阶段的退出标准。

### 阶段 0：基础补齐与路线建立

**建议时间：2～4 周**

#### 学习内容

- Python 项目结构；
- 类型标注；
- 虚拟环境和依赖管理；
- HTTP 与 REST API；
- JSON、Pydantic；
- 异步编程基础；
- Git、Linux、Docker；
- LLM API 基本调用；
- Token、上下文、温度、结构化输出。

同时长期补充：

- 数据结构与算法；
- 操作系统；
- 计算机网络；
- 数据库；
- 并发基础。

#### 阶段项目

完成一个命令行程序：

```text
用户输入问题
→ 调用模型
→ 返回结构化 JSON
→ 记录 Token、耗时和错误
```

#### 阶段目标

建立可复用的 Python 项目基础，理解一次模型请求从输入到输出的完整过程。

#### 退出标准

- 能独立创建 Python 项目；
- 能调用至少一个模型 API；
- 能管理密钥和环境变量；
- 能处理超时、异常和无效响应；
- 能记录耗时和 Token 使用；
- 能解释一次模型请求完整经过了什么。

---

### 阶段 1：Tool Calling 与单 Agent

**建议时间：4～6 周**

#### 学习内容

- Function Calling；
- Tool Schema；
- Agent Loop；
- ReAct 思想；
- Structured Output；
- Session；
- Prompt 与上下文管理；
- 工具选择；
- 重试和最大步数；
- 基础 tracing。

#### 阶段项目

完成第一版：

> **开源项目启动排障 Agent**

工具包括：

```text
scan_project
read_file
read_log
detect_runtime
check_port
generate_start_command
```

第一版只分析问题并生成建议命令，不直接执行危险操作。

#### 阶段目标

Agent 可以根据用户问题和日志主动选择正确工具，而不是由用户手动指定每一步调用哪个函数。

#### 退出标准

- 不使用 Agent 框架也能写出最小 Agent Loop；
- 能解释工具定义如何交给模型；
- 能限制最大调用次数；
- 能记录每一步 Tool Call；
- 能处理工具失败、参数错误和超时；
- 能说明模型决策与程序控制各自负责什么。

---

### 阶段 2：RAG 与知识库

**建议时间：6～8 周**

#### 学习内容

- Embedding；
- 文档切分；
- 向量检索；
- 关键词检索；
- 混合检索；
- Metadata Filter；
- Rerank；
- Query Rewrite；
- 引用来源；
- RAG Evaluation；
- 文档更新和索引重建。

#### 阶段项目

为排障 Agent 增加：

```text
Spring Boot 文档库
Vue/Vite 文档库
Docker 错误案例库
项目 README 检索
历史故障案例检索
```

#### 阶段目标

Agent 不只依靠模型猜测，而是能够从指定资料中检索证据、生成答案并标注来源。

#### 退出标准

- 能说明 RAG 和模型训练的区别；
- 能解释召回失败和生成失败；
- 能建立至少 30～50 条测试问题；
- 能测量检索是否命中正确资料；
- 回答能够显示来源；
- 能分析切分方式、召回数量和重排策略对效果的影响。

---

### 阶段 3：状态化工作流与 LangGraph

**建议时间：6～8 周**

#### 学习内容

- State；
- Node 与 Edge；
- 条件分支；
- Checkpoint；

## 8. 阶段掌握度评估机制

每个阶段都建立五类考核，避免只用“看完课程”衡量进度。

### 8.1 理解考核

不看资料解释：

```text
Agent 和普通聊天机器人有什么区别？
Workflow 和 Agent 有什么区别？
Tool Calling 的控制权在哪里？
RAG 哪一步最容易失败？
为什么需要 Human-in-the-loop？
MCP 解决了什么问题？
```

### 8.2 编码考核

从空目录开始完成最小实现，不复制教程，不依赖已有项目骨架。

### 8.3 修改考核

给原项目加入新的约束，例如：

```text
增加工具权限
增加超时
增加用户审批
增加任务恢复
更换模型供应商
更换向量数据库
```

### 8.4 故障考核

主动制造并处理：

```text
API 超时
JSON 格式错误
Agent 无限循环
错误工具选择
检索无结果
数据库断开
任务中途重启
```

### 8.5 数据考核

至少记录：

```text
任务完成率
工具选择正确率
平均执行步骤
检索命中率
平均延迟
Token 消耗
失败类型分布
```

没有测试集的 Agent 项目，很容易停留在“手工试了几个问题，感觉还不错”的阶段。

### 8.6 建议的能力等级

可以在 `SKILL_MATRIX.csv` 中给每项能力标记等级：

| 等级 | 标准 |
| --- | --- |
| 0 | 未接触 |
| 1 | 能描述概念 |
| 2 | 能跟随资料实现 |
| 3 | 能独立实现并修改 |
| 4 | 能调试、测试和评估 |
| 5 | 能工程化部署，并解释关键权衡 |

每次升级等级都应附上证据链接，例如代码、实验报告、测试结果或演示视频。

## 9. 与 AI 助手的协作方式

协作分工可以设计为：

- 你负责收集原始资料、岗位信息、代码、日志和个人学习反馈；
- AI 助手负责把材料加工成结构化、可学习、可执行和可评估的内容。

### 9.1 资料输入后的处理流程

```text
读取资料
→ 判断可靠性和适用阶段
→ 提取核心概念
→ 关联岗位能力
→ 安排阅读顺序
→ 生成理解问题
→ 设计编码实验
→ 检查理解
→ 检查代码实现
→ 设计测试集
→ 更新路线
```

### 9.2 处理一批学习资料的推荐指令

```text
请读取这批资料，并完成：

1. 判断每份资料适合哪个学习阶段；
2. 提取共同知识点和相互冲突的观点；
3. 区分必须掌握、了解即可和暂时跳过的内容；
4. 给出阅读顺序；
5. 为每个核心知识点设计一个最小编码实验；
6. 更新当前能力矩阵；
7. 列出资料仍未回答的问题。
```

与其只说“帮我总结”，这组任务更容易把资料转化为实际学习行动。

### 9.3 处理岗位资料的推荐指令

每积累一批岗位，可以要求：

```text
1. 提取所有岗位要求；
2. 合并同义技能；
3. 区分基础能力、Agent 能力、算法能力和工程能力；
4. 统计频率和权重；
5. 对比当前学习路线；
6. 找出缺失能力；
7. 调整未来两个月的优先级。
```

### 9.4 处理项目资料的推荐指令

把 README、代码、日志和测试结果一起提供，可以要求：

```text
1. 判断项目实际使用了哪些 Agent 能力；
2. 找出“表面使用框架但没有真正实现”的部分；
3. 设计故障测试；
4. 设计 Evaluation 数据集；
5. 补充项目架构和 README；
6. 把项目能力映射到岗位要求。
```

### 9.5 建议的定期协作节奏

- 每收集 10～15 个岗位：更新一次技能统计；
- 每完成一个主题：进行一次五类考核；
- 每周：更新 `LEARNING_STATE.md` 和周复盘；
- 每月：重新对照岗位需求并调整下月优先级；
- 每完成一个项目里程碑：补充测试、失败案例和作品集材料。

## 10. 时间分配建议

每周可以按照以下比例分配：

```text
60%：编码实验和项目
20%：计算机基础与后端工程
10%：官方资料阅读
10%：岗位分析和学习复盘
```

不要让资料阅读占据一半以上的时间。Agent 是高度工程化的方向，很多理解只能在工具调用失败、检索错误、状态丢失和任务恢复过程中真正建立起来。

每周最好产出一个可以检查的结果：

```text
一个功能
一组测试
一份实验报告
一个错误分析
一张架构图
一次项目迭代
```

不要只用“看了多少小时视频”作为进度指标。

## 12. 最终执行原则

1. **不要把“收集足够资料”设为开始学习的前置条件。** 资料永远收集不完，收集、岗位分析和编码实验应该并行推进。
2. **用能力而不是框架名称组织路线。** 框架会变化，Tool Calling、状态管理、检索、评估和权限控制等能力更稳定。
3. **每个阶段都必须有退出标准。** 时间到了但能力未达到，不算完成；提前达到标准则可以进入下一阶段。
4. **所有学习都尽量形成证据。** 代码、测试、指标、报告、失败案例和演示都比“看完了”更可靠。
5. **优先做一个持续演进的主项目。** 让同一个排障 Agent 逐步加入 Tool Calling、RAG、工作流、MCP 和工程化能力，更容易形成有深度的作品。
6. **定期用岗位库校准路线，但不要被单个岗位牵着走。** 关注稳定的能力交集，而不是追逐每个新框架。
7. **失败案例是学习成果的一部分。** 能解释为什么失败、如何定位、怎样验证修复，是 Agent 工程能力的重要组成。
8. **把评估前置。** 在项目早期就建立测试问题和基础指标，不要等功能全部完成后才考虑效果是否可靠。

最终目标不是“学完 Agent”，而是形成一套可以持续运转的个人能力增长系统：

```text
企业需求提供方向
资料提供知识输入
实验建立真实理解
项目形成能力证据
评估暴露薄弱环节
复盘推动下一轮学习
```
